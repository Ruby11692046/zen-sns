# SNS 要件定義

## ユーザーストーリー

> 学生として私は、Slack（学校として使用しているSNS）ではできない境界線の曖昧な棲み分けをしたい。  
> なぜなら誰でも楽しめる雑談とニッチな趣味仲間の会話の共有のどちらもしたいから。

## 機能一覧

### 優先度凡例

| 記号 | 優先度 | 意味 |
|------|--------|------|
| 🔴 | Must | これがないとサービスとして成立しない |
| 🟡 | Should | できれば入れたい・後回しもOK |
| 🟢 | Could | 余裕があれば実装 |
| ⚪ | Won't | 今回はスコープ外 |

## その他

**意図的にスコープ外とした機能：**
| 機能 | 理由 |
|------|------|
| DM（ダイレクトメッセージ） | Slackで行う想定のため不要 |
| 鍵垢 | Slackで行う想定のため不要 |
| 投稿編集 | 将来対応候補（Won't） |
| 異議申し立てフォーム自作 | Googleフォームで十分なため外部サービスに委譲 |


**処理フロー：**
1. フロントエンドがコマンドフラグを付けて送信
2. バックエンドがメッセージ先頭の `/` を検出してコマンド判定
3. 管理者リストと照合し、管理者であればコマンドとして処理
4. 管理者でなければ通常の投稿として保存・配信

**備考：**
- コマンドはバックエンドのコマンドハンドラーモジュールにまとめて、随時追加できる構造にする
- ミュート状態はDBに保存する（再起動後も維持）
- 異議申し立てはGoogleフォームへの誘導リンクをミュートユーザーに表示する
